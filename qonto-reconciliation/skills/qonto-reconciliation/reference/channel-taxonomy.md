# Channel taxonomy — where a receipt/invoice actually lives

Distilled from a real multi-session reconciliation effort (a few hundred transactions, dozens of distinct suppliers) that reached ~96% document coverage by value. Company names below are cited as concrete, reusable examples of *how a given SaaS/portal category behaves* — any Qonto customer using the same widely-used tools will hit the same mechanics. Try channels in this order (cheapest/most reliable first). The Researcher should identify which channel applies to a given supplier, execute it, and record the exact working path in `businesses.retrieval_pattern` regardless of outcome (a documented failure is as valuable as a documented success — it stops the next run from repeating it).

## Channel A — email attachment (most common, often 40%+ of volume)
The document is a direct attachment on an email from the supplier. Prefer whatever mail access gives you the raw attachment bytes directly (e.g. a Gmail API `messages.attachments.get`-style call) over a generic mail connector/MCP that only exposes attachment ids/metadata without bytes — this is a common real limitation: an MCP mail connector can search threads and list `attachmentIds`, but not export the bytes. When bytes aren't exportable, fall back to a browser: open the specific message (a direct message-id URL if the mail provider supports it), use the message's own "download attachment" control — **not** a click on the attachment thumbnail/chip itself, which usually only opens a preview — and let the file land in the normal downloads location.

Watch for attachments served as `application/octet-stream` instead of `application/pdf` (seen from at least one airline's invoicing system): the obvious big "download" button/chip click can be inert on this MIME type; look for a small secondary hover icon (a down-arrow that appears on hovering the attachment chip) — that's the one that actually triggers the download for non-standard MIME types.

Recurring real examples of this channel: monthly cloud/workspace invoices (e.g. Google Workspace) landing as a numbered PDF attachment around the 1st of the month; co-working space / chamber-of-commerce-style membership invoices as a named PDF (`INV-*.pdf`, `A2026-*.pdf`-shaped) — the latter is sometimes a scanned image with no extractable text, which pushes matching to metadata rather than content (see `matching-heuristics.md`).

## Channel B — hosted link inside an email (Stripe-billed SaaS and similar)
Billing emails from Stripe-integrated SaaS (`invoice+statements+acct_<id>@stripe.com`-shaped senders) contain a direct link to a hosted PDF (`pay.stripe.com/invoice/.../pdf`) that can be fetched **without authentication** — no browser needed, a plain `curl`/fetch works. Try this first, it's the cheapest possible path.

**These links expire — observed at roughly 3 weeks**, after which the URL returns an HTTP error (e.g. 410) or a small "this URL has expired" HTML page instead of the PDF. The fallback is always channel A: the *same email* almost always also carries the PDF as a plain attachment, and a companion "Receipt" email (payment confirmation) typically arrives separately a few days after the invoice email. Treat "hosted link expired" as a normal, expected branch to channel A, not a dead end — and note that `dashboard.stripe.com/receipts/...` links require the account's own session (return a login page if fetched directly) while `files.stripe.com` links typically only serve logos/assets, not the invoice itself — don't rely on either as a fetch target.

Products billed this way in practice: most modern AI/dev-tool SaaS (e.g. Anthropic, Krea-style AI tools), which is the majority of "Channel B" cases.

## Channel C — web portal (the hard case, several sub-patterns)

**C1 — portal navigable in an authenticated browser session.** If a browser automation tool is available and already has an authenticated session with the supplier, navigate to the billing/invoices section and download directly. Look specifically for a "download by period" or bulk-export feature (e.g. a telecom operator's customer portal offering "download invoices for period: last month / quarter / year / custom", which also emails a notification when the batch is ready) — much more efficient than fetching one invoice at a time.

**C2 — one-shot account setting that permanently upgrades the channel.** Many portals (the same telecom-operator example above is a real case) have a "send invoice by email" toggle in account settings. When found, flag it as a suggested one-time setup action — enabling it permanently converts that supplier from channel C to channel A for every future invoice. This is the single highest-leverage move available: always check for it on any new C1 portal.

**The generic "PDF gated by session" pattern** — the single most broadly useful technique discovered, applicable to most SaaS/marketplace portals where the invoice lives behind a session:

*Problem*: the invoice URL (typically shaped like `.../documents/download/<uuid>/invoice.pdf`, or a hosted Stripe portal URL) requires session cookies a bare `curl` doesn't have (returns 403 or a login redirect), and the mail connector doesn't export attachment bytes either. A browser's native PDF viewer is also usually not scriptable for this purpose (attempting to run script against a PDF-viewer tab typically errors, e.g. "cannot attach to this target").

*Solution* (3 conditions):
1. Be on a normal HTML page of the *same origin* as the PDF — not the PDF viewer itself.
2. Run, in-page: `fetch(url, {credentials: 'include'}) → blob → <a download> → click()`. This inherits the page's own session cookies where an out-of-browser `curl` cannot. The file lands in the normal downloads location.
3. Only report a short status back (e.g. "downloaded"), never the signed URL or raw bytes as tool output — some harnesses redact tool output containing base64 blobs or signed query strings as a privacy safeguard (a real observed behavior: output gets replaced with something like `[BLOCKED: Cookie/query string data]`). The side effect (the file landing on disk) still happens even when the output text is redacted — design the action so you don't depend on reading that output back, just confirm the file exists afterward.

Real examples this exact pattern resolved: a large marketplace's multi-vendor order invoices (each vendor/seller in a multi-seller order produces its own separate "Factura" link, e.g. Amazon — one order can require attaching 2+ separate invoice PDFs to the same single bank transaction); an ad platform's billing hub receipts (e.g. Meta Ads' `billing_hub/payment_activity`, see next point for the accompanying date-picker trick).

**Capricious UI filters (e.g. a fragile date-range picker)**: if the filter state is reflected in the URL as a query parameter, prefer editing the URL parameter directly over fighting the widget. Concretely proven on an ad platform's billing activity page: instead of wrestling the date-picker widget, edit the `date=<unixStart>_<unixEnd>` query parameter directly in the URL, reload, then use the per-row "download PDF" link with the fetch→blob technique above.

**A portal action that opens an entirely new, uncontrolled browser tab/window.** Common with "manage my subscription" buttons that hand off to a payment processor's own hosted portal (observed concretely with an AI chat product whose "manage subscription" opens its Stripe billing portal in a brand-new tab outside the browser-automation tool's controlled tab group — the agent loses the ability to act in that tab). Treat this as a distinct blocker (`window_open_escapes_control`) rather than silently failing — it needs either a browser tool that controls all tabs/windows, or one human click to finish in that tab.

**Print-only receipts.** Some portals (a well-known domain registrar's receipt page is a clean real example) render a receipt as an HTML modal whose only "download" affordance is the browser's native print-to-PDF dialog — not a real PDF link. That native dialog is chrome-of-the-browser, not part of the page, and is generally not scriptable by a browser-automation extension; a plain on-screen screenshot capture is also typically not treated as an uploadable file by that same class of tool (confirmed a dead end, not a workaround). This is the clearest real case where a fuller, real browser-automation engine (one that can render a page to PDF server-side, e.g. a Playwright-style `page.pdf()` call) resolves it cleanly where an extension-based approach cannot — flag `print_only_needs_fuller_browser` rather than attempting a screenshot workaround that won't actually be uploadable. The same category (print-only / no real PDF link) also shows up on some subscription-payment confirmation pages (e.g. `pay.google.com` activity views).

**Login-gated portals with no existing session.** Several categories of tool default to no persistent browser session even when other portals are logged in — dev-tool SaaS with their own separate auth domain, note-taking/wiki SaaS that's normally used via a desktop app rather than the browser, outreach/podcast-hosting SaaS, etc. If there is genuinely no authenticated session and no credential to establish one, this is a one-shot human setup blocker (`login_required`) — not a dead end. Once a human logs in once in the browser the automation tool controls, the supplier permanently moves to channel C1/auto from then on.

**Billing email routed to the wrong inbox.** Worth checking explicitly: some SaaS accounts end up billed against a personal email address rather than the company one (e.g. because sign-up happened from a personal account originally), meaning the company's mail search for `from:<billing sender>` never finds anything even though the receipts objectively exist — they're just landing somewhere else entirely. When a Channel A/B search comes up completely empty for a supplier that's clearly being charged, check whether the account itself might be tied to a different, non-company email before concluding the supplier needs a portal visit.

## Channel D — human supplier, no invoice exists yet
Freelancers/small vendors who don't proactively send invoices (event/production service providers, small studios, individual contractors): the invoice must be requested. Draft the request (email or message) but never send it (see guardrails) — leave it ready for the operator to review and send. Once sent (by the operator) and a reply arrives, it becomes channel A on the next sweep.

## Channel E — local deposit / photo
Physical receipts (restaurants, taxis, small purchases): expect the operator to drop these into a conventional local folder (e.g. `~/Downloads/receipts-inbox/` or an equivalent the operator names). Match by amount + date proximity. Also check any other locations the operator names (a shared drive folder, the desktop, etc.).

## Channel F — request-by-form, delivery DEFERRED by email
Some suppliers have no downloadable document at all: filling out a request form is the only interface, and the actual invoice arrives by email later. A concrete real example: an airline's invoice-request flow — locate the booking via a "retrieve my booking" form (booking reference + email), open the account's invoice-request section, and submit a request form. The disclaimer text may claim a specific SLA (e.g. "within 48 hours") or claim it only ships after some triggering event (e.g. "after your last flight segment") — in practice the invoice arrived within a few hours in one observed case, well before the theoretical trigger event. **Don't assume the stated SLA/trigger is exact — poll rather than wait for a theoretical date.**

A recurring wrinkle on this channel: the request form may be shaped for a private individual (asks for first/last name) with no explicit "I'm a business" toggle, yet still accepts a company's tax ID in the personal ID field (e.g. a national tax ID field accepting a company registration number) — a workable mapping is company legal name split across name/surname fields with the company tax ID in the personal-ID field, but this is exactly the kind of legal/fiscal-identity judgment call that must be shown to the operator before treating the form as ready to submit (never assume this mapping silently). Fill the form out fully but never submit it autonomously (see guardrails) — leave it ready, precisely described, for the operator to submit. Also: don't guess at URLs for this kind of flow — a guessed deep-link (e.g. `/manage-your-booking`) can 404 after a site restructure; always navigate from the real, current entry point. Once actually submitted, treat the payment as `awaiting_async`; the next routine sweep of channel A will pick up the reply.

## Channel for credits — payout/settlement reports (not a per-transaction document)
Incoming payment-processor payouts (e.g. marketplace/ticketing-platform settlements routed through Stripe or similar) are typically **not** backed by a single per-transaction invoice. The correct supporting document is a periodic payout/reconciliation report from the processor's dashboard (e.g. Stripe Dashboard → Reports → a "payout reconciliation" style report), covering a date range that includes the payout. Don't search for a 1:1 document for these — search for the report covering the right period, and match by `period_covered_from/to` overlapping the transaction's settlement date rather than an exact amount (a payout report also nets fees, so the report total may differ from any single transaction). The same "fees are netted, not invoiced per-transaction" situation applies to a platform's own processing fees on such an account — the fiscal document is a periodic fee statement, not a per-debit invoice, and is sometimes described in the platform's own UI as fees having been "moved into/absorbed by the plan" rather than billed individually.

## General web search
Use as a last resort when the supplier/portal itself is unknown (to find the right billing portal URL, support docs, etc.) — never to guess a document's URL directly. If a specific URL 404s, that's a signal the site has been restructured; navigate from the real entry point instead of guessing further paths.
